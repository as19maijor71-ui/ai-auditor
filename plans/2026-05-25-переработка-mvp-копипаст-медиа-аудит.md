# План: переработка MVP под копипаст страницы и медиа-аудит (25.05.2026)

## Цель
Переработать MVP AI Аудитора WB/Ozon так, чтобы главный сценарий был простым для проверки спроса:

1. Пользователь копирует страницу карточки WB/Ozon целиком (`Ctrl+A` -> `Ctrl+C`).
2. Вставляет текст в Telegram-бот.
3. Получает быстрый аудит по текстовым данным.
4. По желанию добавляет фото галереи и описание видео.
5. Получает итоговый сохраняемый отчёт.

Это временная MVP-схема для проверки спроса. Если будет отклик и платящие пользователи, следующий этап — браузерное расширение с аудитом в один клик.

## Почему меняем текущую схему
Старая схема опиралась на автозагрузку по ссылке и экспорт XLSX/CSV:

- WB/Ozon часто блокируют серверную загрузку страниц.
- Экспорт из личного кабинета даёт неполные данные и не показывает карточку глазами покупателя.
- Ручной 5-шаговый сбор данных слишком тяжёлый для пользователя.
- Полный аудит фото, видео, отзывов и конкурентной полки невозможен только по Excel.

Новая схема использует то, что реально даёт больше данных без сложной интеграции: полный копипаст страницы карточки из браузера.

## Архитектурная переработка

### Было
Старый поток:

```text
URL/Excel -> парсер -> сырой текст -> один большой промт -> OpenRouter/YandexGPT -> AuditReport -> 5 сообщений
```

Проблемы старого потока:

- URL-загрузка ненадёжна из-за блокировок WB/Ozon.
- Excel не содержит достаточно данных для полного аудита.
- AI получает сырой шумный текст и вынужден сам отделять товар от меню, футера и рекомендаций.
- Один промт пытается сделать всё сразу: парсинг, анализ, медиа, конкурентов, отчёт.
- OpenRouter и YandexGPT не дают нужной стабильности для MVP.

### Станет
Новый поток:

```text
Копипаст страницы частями или .txt файл
  -> PasteCollector
  -> PasteNormalizer
  -> MarketplaceCardSnapshot + LocalAuditFacts
  -> Gemini API
  -> AuditReport
  -> ReportExporter
  -> Telegram-сообщение + .txt файл
  -> опционально: фото + описание видео
  -> обновлённый AuditReport с медиа-блоком
```

Главное изменение: AI больше не получает просто «мусорный текст страницы». Перед AI появляется слой нормализации и локальных правил. AI получает структурированный вход и формирует отчёт на основе уже извлечённых фактов.

### MVP-слои
- [x] `PasteCollector` — принимает копипаст частями и `.txt` файл, потому что одна страница может не влезть в одно сообщение Telegram.
- [x] `PasteNormalizer` — чистит копипаст страницы от навигации, футера, повторов и мусора.
- [x] `MarketplaceCardParser` — извлекает данные WB/Ozon в единую структуру.
- [x] `LocalAuditRules` — считает факты без AI: длину заголовка, повторы слов, наличие цены, отзывов, характеристик, описание, конкурентные цены.
- [x] `AuditRunner` — вызывает Gemini, делает retry и fallback в OpenAI только при отказе основного провайдера.
- [x] `ReportExporter` — собирает сохраняемый `.txt` отчёт из `AuditReport`.

### Не делать в MVP без доказанной необходимости
- `AuditOrchestrator` как отдельный сложный слой.
- `ReportComposer` как отдельный сложный слой.
- Отдельные input/output DTO на каждый шаг, если хватает `MarketplaceCardSnapshot`, `LocalAuditFacts` и существующего `AuditReport`.

## Модели данных для MVP

Правило: не плодить DTO ради архитектуры. В MVP реализуются только модели, которые прямо нужны для парсинга, локальных фактов, AI-ответа и сохранения отчёта.

### Используем существующее
- [x] `AuditItem` и `AuditReport` остаются основной моделью AI-ответа.
- [x] Сохраняемый `.txt` отчёт собирается из `AuditReport`, без отдельной модели `FinalAuditReport` на первом шаге.

## Новые AI-модели и провайдеры

### Было
- Основной провайдер: OpenRouter.
- Основная модель: DeepSeek-R1 через OpenRouter.
- Fallback: YandexGPT.
- Vision/OCR: Gemini через OpenRouter.

### Станет
- Основной провайдер для MVP: прямой Gemini API.
- Основная модель текстового аудита: `gemini-2.5-flash`.
- Модель для OCR/медиа-анализа: сначала `gemini-2.5-flash-lite`; если на тестовых фото даёт слабое описание, фиксированно переключить на `gemini-2.5-flash`.
- Fallback для финального текстового отчёта: прямой OpenAI API, модель `gpt-5-mini`.
- OpenRouter остаётся только как аварийный/отладочный вариант, не основной путь.
- YandexGPT убирается из аудита.

### Почему так
- Gemini лучше подходит для MVP, потому что закрывает и текст, и изображения.
- Прямой API убирает лишнюю точку отказа в виде агрегатора.
- OpenAI fallback нужен для стабильного структурированного финального отчёта.
- YandexGPT не подходит как fallback для такого аудита: слабее по конкретике и стабильности JSON.

## Новые промты
- [x] `QUICK_TEXT_AUDIT_PROMPT` — быстрый аудит по `MarketplaceCardSnapshot` + `LocalAuditFacts`.
- [x] `MEDIA_AUDIT_PROMPT` — аудит фото и описаний видео, только если в этой итерации включаем медиа-аудит.
- [x] `FINAL_REPORT_PROMPT` не делать на первом шаге: сначала собирать `.txt` отчёт детерминированным форматтером из `AuditReport`.
- [x] Для нового quick path убрать один монолитный промт, который пытается сделать всё сразу.

## Deliverable
Обновлённый Telegram-бот, в котором:

- главный вход — вставка полного текста страницы WB/Ozon частями или загрузка `.txt`;
- быстрый аудит запускается сразу после текста;
- визуальное расширение доступно как дополнительный шаг;
- видео описывается текстом по шаблону;
- итоговый отчёт выдаётся в удобном для сохранения виде;
- Excel/CSV и ссылка не развиваются в этой итерации: существующие сценарии не ломать, но новый код под них не писать.

## Что считаем «сделано»
- Для 10 тестовых копипастов (5 WB, 5 Ozon) бот без AI корректно определяет площадку и название товара в 10/10 случаях.
- Для тех же 10 копипастов бот без AI извлекает цену, рейтинг или количество отзывов минимум в 8/10 случаях.
- Быстрый аудит запускается после подтверждения пользователя и возвращает `AuditReport` с минимум 5 секциями: title, price/competitors, description, seo, reviews/risks.
- В отчёте всегда есть блок «Что не проверено»: фото, инфографика, видео, порядок галереи, если медиа не добавлены.
- Бот отправляет итоговый `.txt` файл и короткое Telegram-сообщение с 3 главными действиями.
- Лимит бесплатных аудитов проверяется до вызова AI и до OCR/визуального анализа.
- Существующие команды `/start`, `/help`, `/stats`, `/myid` не ломаются.

## Риски
- Копипаст из мобильного приложения может отличаться от копипаста из браузера.
- В тексте страницы много мусора, рекомендаций и футера.
- Пользователь может вставить не всю страницу.
- Telegram может неудобно принимать очень длинный текст.
- Фото без порядка ухудшают медиа-аудит.
- AI может принять блок рекомендаций за данные основного товара, если парсер плохо отделит секции.
- OCR/визуальный анализ 8 фото может стать основной себестоимостью аудита.
- Старые FSM-состояния и callback-кнопки могут конфликтовать с новым сценарием.
- Копипаст может содержать персональные данные доставки/адреса пользователя, их нужно удалять в нормализации.
- Блок «Подобрали для вас» может содержать не конкурентов, а товары из другой категории.
- Пользователь может отправить чужой товар или подборку, и отчёт будет выглядеть уверенно при плохом входе.
