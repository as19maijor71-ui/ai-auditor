# Главный кирпич: AuditReport

## Гипотеза

Весь продукт держится на одной сущности — **AuditReport**. Это результат аудита карточки. Если эта сущность спроектирована правильно, всё остальное (вывод, хранение, статистика) строится вокруг неё.

## Pydantic-модель

```python
class AuditItem(BaseModel):
    section: str          # title / photos / description / seo / competitors
    priority: str         # red / yellow / green
    finding: str          # что найдено (1 предложение)
    recommendation: str   # что исправить (конкретно)
    why: str              # почему это важно

class AuditReport(BaseModel):
    url: str
    platform: str         # wb / ozon
    product_name: str
    overall_score: int    # 0-100
    items: list[AuditItem]
    summary: str
    competitor_insight: str
```

## Инварианты

1. **5 секций обязательно.** Даже если проблем нет — «всё хорошо»
2. **Приоритеты только red/yellow/green.**
3. **Overall score 0-100.**
4. **Не более 3 red-проблем.** Если больше — выбираем самые критичные
5. **Каждый finding = 1 предложение.** Читаемо в Telegram

## Что выигрываю
- Одна сущность — одна таблица — простые запросы
- Pydantic валидация на входе и выходе
- Лёгкая сериализация для хранения в SQLite

## Что теряю
- Нельзя хранить сырой ответ AI для отладки
